Overview

What is OpenSpec?

A framework that puts software specifications inside your repo — so AI coding assistants stop guessing and start understanding your system.

💡
One sentence: Instead of re-explaining your architecture every AI session, OpenSpec stores requirements, decisions, and architecture in Git — permanently, alongside your code.
The core shift
✕ Prompt-only today
Prompt → AI guesses requirements → Generates code → You fix inconsistencies → Repeat next session
✓ OpenSpec workflow
Spec → AI understands requirements → Generates code → Specs stay in Git → Next session: same context
Best for
🏗️

Large codebases

Context grows too big for one AI session. Specs keep the whole picture available.

👥

Teams

Multiple developers need shared, consistent understanding that doesn't live in anyone's head.

🕰️

Long-lived projects

Decisions made six months ago still matter. Archives make that history searchable.

🤖

Heavy AI usage

The more you use Cursor, Claude Code, or Copilot, the more persistent context pays off.

Architecture

Core concepts

Five building blocks. Each has one job. Together they form the complete OpenSpec system.

1 — Specs (source of truth)

The official, current description of what your system does. Lives in openspec/specs/. AI reads this before implementing anything.

## Requirement: Session Expiration The system SHALL expire inactive sessions after 24 hours. ### Scenario: Auto-logout GIVEN a user is authenticated WHEN 24 hours pass without activity THEN invalidate the session token
2 — Changes (feature workspaces)

Every feature or fix lives in its own isolated folder. One change = one logical feature.

openspec/changes/add-oauth-login/
├── proposal.md← why this feature exists
├── design.md← technical architecture
├── tasks.md← checklist for AI to execute
└── specs/← what requirements change
3 — Tasks (AI execution steps)

Atomic implementation steps. AI executes them one-by-one. Smaller = better output quality.

- [ ] Create /auth/google/callback endpoint - [ ] Add passport-google-oauth20 library - [ ] Implement token exchange logic - [ ] Store session in httpOnly cookie - [ ] Add CSRF state param validation
4 — Delta specs (change tracking)

What changed in this feature — just the diff, not the whole spec rewritten.

- REMOVED Requirement: Password-only auth + ADDED Requirement: Google OAuth Login MODIFIED Requirement: Session Management - Sessions expire after 24 hours (fixed) + Sessions support configurable expiration (default: 24h)
5 — Archives (permanent history)

Completed changes move here. Future developers and AI sessions can trace every decision.

openspec/archive/add-oauth-login/← preserved forever
├── proposal.md
├── design.md
├── tasks.md
└── specs/
Architecture

Repo structure

One directory added to your project. Three folders. Everything in Git — nothing external.

openspec/
├── specs/← current system truth (always up to date)
│   ├── auth/spec.md
│   ├── payments/spec.md
│   └── notifications/spec.md
│
├── changes/← active work in progress
│   └── add-oauth-login/
│       ├── proposal.md
│       ├── design.md
│       ├── tasks.md
│       └── specs/
│
└── archive/← completed features (never delete)
    └── add-dark-mode/
specs/

Live, accurate description of what your system does right now. AI reads this before every implementation.

changes/

Work-in-progress feature workspaces. Each has its own proposal, design, and tasks — isolated per feature.

archive/

Full history of every completed feature. Priceless for new team members and future AI sessions.

⚠️
Never gitignore these folders. Specs, changes, and archives are first-class code — commit them exactly like source files.

Common errors
command not found

Global npm path isn't in your PATH. Fix: use NVM — it handles PATH automatically.

EACCES permission denied

Prefix with sudo, or switch to NVM so global installs never need elevated permissions.

Setup

Project setup

Initialize once. Adopt incrementally. You don't need to spec your whole codebase on day one.

Initialize
cd my-project openspec init # Creates: openspec/specs/ openspec/changes/ openspec/archive/
For existing projects — start with one module
💡
Brownfield tip: Start with the module that causes the most AI confusion — usually auth or payments. One good spec immediately pays off. Expand from there.
🔐

Start: Auth

High security impact, lots of edge cases — the perfect first spec that immediately improves AI output.

💳

Then: Payments

Complex business logic, many failure modes. Spec-first prevents AI missing critical paths.

🔔

Then: Notifications

Multiple channels, delivery guarantees — benefits greatly from structured scenarios.

📦

Grow from there

APIs → frontend → infrastructure. No rush — each spec adds value immediately.

Git best practices
📦

Commit specs with code

Always include spec changes in the same commit as the code. Reviewers see the "why" alongside the "what."

👁️

Review specs in PRs

Review spec changes before code changes. Understanding requirements first makes code review much faster.

🗄️

Never delete archives

Archives are institutional memory. Future AI sessions depend on them to understand historical decisions.

Workflow

Writing specs

Good specs are short, testable, and unambiguous. Here's every file type with a real example.

## Requirement: User Login The system SHALL allow users to log in using email and password. ## Acceptance Criteria - Valid credentials authenticate the user - Invalid password returns 401 (without revealing if email exists) - 5 failed attempts triggers a 15-minute lockout - Sessions expire after 24 hours of inactivity
Use SHALL for mandatory behavior. One requirement = one behavior. Each criterion must be directly testable — if you can't write a test for it, rewrite it.
### Scenario: Successful login GIVEN a registered user with a valid account WHEN the user submits correct email and password THEN return a session token and 200 OK ### Scenario: Brute force protection GIVEN a user has failed login 5 times in 15 minutes WHEN they attempt another login THEN return 429 Too Many Requests AND include a Retry-After header
GIVEN / WHEN / THEN maps directly to test cases. Write at least one failure scenario — it forces you to think through edge cases before they hit production.
# Proposal: Google OAuth Login ## Problem Support receives 40+ password-reset tickets per week. Signup conversion sits at 48% — below the industry average of 65%. ## Solution Add Google OAuth as an alternative path. Existing email/password login remains fully unchanged. ## Expected impact - Signup abandonment: -30% - Password-reset tickets: near-zero for OAuth users - Onboarding time: 3 min → 15 seconds ## Out of scope - Apple/GitHub OAuth (separate future change) - Mobile app OAuth (separate future change)
Keep it short: problem, solution, numbers, scope. Future AI sessions use this to understand business intent — not just what to build, but why it matters.
# Tasks — add-oauth-login Backend - [ ] Create GET /auth/google → redirect to Google consent - [ ] Create GET /auth/google/callback → exchange code for tokens - [ ] Implement state param (CSRF protection) - [ ] Store session in httpOnly, Secure, SameSite cookie - [ ] Unit tests: callback handler, token exchange, error states Frontend - [ ] Add "Continue with Google" button to login page - [ ] Show loading state during OAuth redirect - [ ] Handle error query params from failed OAuth flow
Tasks are the primary input for /opsx:apply. Keep each task to a single concrete action. "Add Google login" is not a task — it's a feature. Break it down.
# Design: Google OAuth Login ## Auth flow User → GET /auth/google → Google consent screen → GET /auth/google/callback?code=...&state=... → Exchange code for { access_token, id_token } → Verify id_token with Google public keys → Upsert user (email as unique key) → Issue session cookie → redirect /dashboard ## DB changes ALTER TABLE users ADD COLUMN google_id VARCHAR(255) UNIQUE; ALTER TABLE users ADD COLUMN avatar_url TEXT; ## Security decisions - State param: 32-byte random, Redis, 10-min TTL - Cookie: httpOnly, Secure, SameSite=Lax, 24h TTL - Token verify: Google JWKS endpoint, cache keys 1h
Include the flow (even ASCII art), DB changes, and key security decisions. Future AI sessions use this to stay aligned with what was already decided — preventing re-invention.
The golden rule
✕ Vague — AI will guess
"Implement auth with Google"
×AI invents the flow from scratch
×May miss security requirements
×Different result every session
✓ Specific — AI will follow
"Implement Task 3 from tasks.md per the OAuth design in design.md"
✓AI follows the exact defined flow
✓Security decisions are pre-made
✓Same result every session
Concepts

Delta specs

Instead of rewriting whole spec files, OpenSpec tracks exactly what changed — ADDED, MODIFIED, REMOVED, or RENAMED.

Why not just edit the spec directly?
✕ Editing spec directly
×PR diff shows entire spec file changed
×Reviewer can't tell what actually changed
×No history of why it changed
✓ Using delta specs
✓PR diff shows only what changed
✓Reviewer immediately sees the delta
✓History preserved in archive forever
The four operations
＋ ADDED

New requirements, scenarios, or criteria entering the system for the first time.

～ MODIFIED

Existing behavior that's changing. Keeps requirement identity intact while updating its meaning.

－ REMOVED

Retiring behavior. Explicitly marks what's going away so nothing silently disappears.

↔ RENAMED

Same behavior, new name — preserves history under the updated identifier.

Full example
openspec/changes/add-oauth-login/specs/auth.md - REMOVED Requirement: Password-only auth + ADDED Requirement: Google OAuth Login + The system SHALL support authentication via Google OAuth 2.0 MODIFIED Requirement: Session Management - Sessions expire after 24 hours (hardcoded) + Sessions support configurable expiration, default 24h + ADDED Scenario: OAuth callback success + GIVEN a valid authorization code from Google + WHEN the callback endpoint receives it + THEN exchange for tokens and create a session
This delta lives in the change workspace during development. When you run /opsx:archive, it merges into openspec/specs/auth/spec.md automatically — keeping the source of truth current.
